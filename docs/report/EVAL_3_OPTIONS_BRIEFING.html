Team decision doc for which training tracks to run · ETH Robot Learning FS26 Project 1
⚠ Read first: terminology Throughout this document we use the names camera1 and camera2. Their meaning is:
  • camera1 = the actual wrist-camera video recorded during human teleoperation of the SO-101 robot. This is the live robot view that shows the workspace with the three printed celebrity portraits during a demo.
  • camera2 = a constant-frame video stream we constructed where every frame is the same static reference photo of the target celebrity. We use video format because LeRobot's data pipeline expects video, but it is fundamentally just a single still image repeated for the duration of the episode.
This distinction matters for the failure-mode discussion below — camera2 reaches the model as a separate camera observation rather than being inlined inside the language prompt.

§1 · Why v1 failed

SmolVLA-450M was trained for the Eval 3 face-matching task (place coke can on the celebrity portrait specified by name and a reference photo). The smoke-test on the deploy laptop on 2026-05-17 revealed that the trained policy fails: it places the can on the wrong portrait, falling back to positional priors instead of conditioning on identity.

Empirical findings:

  • The pipeline runs end-to-end (no crashes, no OOM, no timing issues).
  • The camera2 reference photo is verified to reach the model (instrumented at prepare_images()).
  • The wrist-camera (camera1) aim is verified identical to training distribution.
  • Failure mode: positional shortcut. Rotating Swift's portrait through left/middle/right physical positions confirms the can lands at the same physical spot regardless of which celebrity is there. The model is not conditioning actions on identity.

Two compounding gaps explain this:

  • (G1) Domain gap. Training-time reference photos (in camera2) are magazine/web style. Eval-day workspace shows paper printouts photographed through a wrist cam. Print-and-Scan face-recognition literature measures this transformation at +5.6–16.0% FMR shift on ArcFace verification.
  • (G2) Representation gap. SmolVLA's vision tower compresses each image into 64 tokens via 2×2 pixel-shuffle (SmolVLM §3.1). For a face occupying ~30% of the frame, that is only ~10–20 identity-bearing tokens. SigLIP was pretrained for image-text alignment, not face-discriminative geometry.

Plus we discovered 6 additional protective mechanisms we under-applied from the very papers we cited (see §2).

§2 · The 8 protective mechanisms

Each mechanism is a research-published intervention that could be added to a training recipe to address a specific failure mode. The architectures (SmolVLA, Pi0.5) can be combined with any subset.

M1 · Frozen 3-layer MLP projector + Backbone2Enc injection

Source. "Don't Blind Your VLA" (arXiv 2510.25616), Tables 5 and 6.

Plain-language description. We need a small module that maps the VLA's internal features into the same space as a teacher encoder's features so we can compare them with a cosine loss. The mapping is a small MLP. The paper proves two non-obvious points:

  1. The MLP must be frozen. Initialise it once, then set requires_grad=False. Otherwise the model "cheats" by adjusting the MLP instead of fixing the VLA's internal state. Table 6: frozen 0.61 vs trainable 0.54 on semantic OOD.
  2. The loss must be injected at a mid-LLM layer (Backbone2Enc), not at the vision encoder output (Enc2Enc). Table 5: Backbone2Enc 0.61 vs Enc2Enc 0.55 on the semantic axis (the vision and execution axes are tied or statistically indistinguishable).

Architecture (code-verified at finetune_align.py:326-338):

nn.Sequential(
    nn.LayerNorm(hidden_size),
    nn.Linear(hidden_size, 2048),
    nn.SiLU(),
    nn.Dropout(0.1),
    nn.Linear(2048, 2048),
    nn.SiLU(),
    nn.Dropout(0.1),
    nn.Linear(2048, z_dim),   # z_dim = teacher feature dim (512 for ArcFace)
)

Cost. ~1 day engineering + 0 extra Brev (folds into M2's training run).

Validation status. ✓ Confirmed verbatim by V2 audit. Table 5 and Table 6 numbers verified; code line numbers verified.

M2 · Cosine alignment loss against a frozen vision teacher

Source. "Don't Blind Your VLA" Equation 9.

Plain-language description. Train the VLA's vision features to match a frozen "teacher" encoder's features. The teacher knows things the VLA doesn't (e.g., ArcFace knows face identity). This is the actual loss term that makes M1 work.

Exact loss:

$$\mathcal{L}_{\text{align}} \;=\; -\frac{1}{k} \sum_{j=1}^{k} \cos\!\big(\,\text{F.normalize}(u_j),\; \text{F.normalize}(z_j)\,\big)$$ $$\mathcal{L}_{\text{total}} \;=\; \mathcal{L}_{\text{action}} \;+\; \lambda \cdot \mathcal{L}_{\text{align}}, \qquad \lambda = 0.2$$

where $u_j$ are projected student patch features, $z_j$ are corresponding teacher patch features, $k$ is the number of aligned patches.

Key details:

  • Cosine, not L2 or InfoNCE. Paper Table 8: cosine 0.61/0.72/0.39 > L2 0.54/0.63/0.34 > InfoNCE 0.57/0.64/0.36. Statistical significance p<0.01 holds on the semantic and vision OOD axes only; on the execution axis p=0.05 (borderline).
  • F.normalize is explicit in the code, then dot product (NOT F.cosine_similarity). Verified at finetune_align.py:420-428.
  • $\lambda = 0.2$, constant from step 0. Verified at finetune_align.py:311.
  • Patch-level, not CLS-level.

Headline empirical impact (Table 1, mean ± SD):

MethodSemantic OODVision OODExecution OOD
Default SFT0.49 ± 0.020.74 ± 0.020.28 ± 0.02
Freeze encoder0.03 ± 0.010.05 ± 0.010.01 ± 0.01
Align (theirs)0.61 ± 0.010.83 ± 0.030.35 ± 0.02

+12 pp on semantic OOD, +9 pp on vision OOD.

Teacher choice for our task. Their best teacher is C-RADIOv3 (a general vision foundation model). For face matching we will use ArcFace (buffalo_l/w600k_r50, 512-dim) instead — task-specific. This is a first-mover combination (no published paper does ArcFace→SigLIP distillation), but the recipe transfers in principle. We will mask-gate the loss to face patches only (via RetinaFace masks computed offline).

Cost. Combined with M1 — ~1 day engineering + ~1.5-3h Brev fine-tune.

Validation status. ✓ Equation 9 confirmed verbatim by V2 audit. Code lines verified. Table numbers verified.

M3 · Stop-gradient between action expert and VLM

Source. Pi0.5 Knowledge Insulation (arXiv 2505.23705), Equations 5 and 6.

Plain-language description. When the action expert reads from the VLM via cross-attention, the gradients flowing backward from the action-prediction loss should NOT update the VLM. The action expert can read VLM features but cannot perturb them. This prevents the VLM's pretrained knowledge from drifting during robot fine-tune.

Exact mechanism. When the action expert attends to the VLM backbone (subscript $b$), both the keys $K_b$ and values $V_b$ are wrapped in $\text{sg}(\cdot)$ (PyTorch's detach()). Self-attention within the VLM ($P_{bb}$) is untouched.

Eq. 5 (attention pattern, action-expert row only shown):

$$\text{softmax}\big(\, Q_a \cdot \text{sg}(K_b)^T \quad Q_a \cdot K_a^T \,\big)$$

Eq. 6 (value computation for the action expert):

$$E_a \;=\; P_{ab} \cdot \text{sg}(V_b) \;+\; P_{aa} \cdot V_a$$

Empirical impact (validated):

  • Frozen VLM = 0% success on items-in-drawer task. Verbatim from §4 line 284: "current VLMs are not pre-trained with robotics data. As a result, their representations, when frozen, are insufficient for training highly performant policies, as we show in our experiments, cf. Fig. 4a and Fig. 8 (0% performance)."
  • Joint training (no stop-gradient, which is our v1 setup) performs significantly worse than KI on action success AND collapses language following. Body text confirms direction; exact bar heights are visual reads from Figs 4a/4b/6a.
  • π₀ needs 7.5× more steps to match KI. Verbatim from Fig. 6 caption: "π₀ trains significantly slower, requiring 7.5 times as many training steps to reach a similar performance."

Cost. Multi-day engineering. Requires modifying SmolVLA's embed_prefix and forward to insert detach() calls on cross-attention K and V from VLM to action expert. Not currently exposed in lerobot 0.5.1.

Validation status. ✓ Eqs. 5-6 confirmed verbatim by V1 audit. Frozen-VLM 0% verbatim. 7.5× speedup verbatim. Specific bar-height percentages flagged as visual reads.

M4 · FAST-token cross-entropy loss through the VLM

Source. Pi0.5-KI §5.1.

Plain-language description. Even though the action expert outputs continuous actions via flow-matching, the VLM is also trained to predict a discretised version of the action sequence as a sequence of language-like tokens (FAST tokens). This gives the VLM a supervised signal that ties its representations to robot actions, without polluting the action expert.

Exact mechanism. Eq. 4 includes an autoregressive token-prediction term:

$$\mathcal{L}_{\text{CO-VLA}} \;=\; \mathbb{E}\!\left[ -\sum_j M_j^\ell \log p_\theta(\hat{\ell}_{j+1} \mid x_{1:j}) \;+\; \alpha \cdot M^{\text{act}} \cdot \|\omega - a_{1:H} - f_\theta^a(a_{1:H}^{\tau,\omega})\|^2 \right]$$

where $\hat{\ell}$ contains "both text (language) tokens and FAST tokenized action tokens". $M^\ell$ and $M^{\text{act}}$ are per-sample masks. With M3's gradient stop in place, $\alpha = 1$.

Empirical impact. Verbatim from §5.1: "The autoregressive objective is only used at training time as a representation learning objective, which enables the model to train much faster." This is the lever that produces the 7.5× speedup from M3.

Cost. Multi-day engineering. SmolVLA's training forward computes only flow-matching MSE — no LM head exposed for next-token CE loss.

Validation status. ✓ §5.1 mechanism confirmed verbatim.

M5 · Web / VQA co-training

Source. Pi0.5 paper (arXiv 2504.16054) + Pi0.5 blog.

Plain-language description. Mix non-robot data into the robot fine-tune to preserve the VLM's pretrained world knowledge. Pi0.5's first-phase training mixture is 97.6% non-mobile-manipulator data — robot from other embodiments, plus image-text pairs from the web.

Empirical impact. Pi0.5 blog verbatim: removing web data drops OOD object recognition from 94% to 74% on the object-into-drawer eval. −20 pp on OOD object recognition — the closest published analog to our face-name binding task.

Cost. Two ways:

  1. Full co-training (proper way): mixed-batch training of robot + VQA data with per-sample loss masking. Blocked by MultiLeRobotDataset = NotImplementedError in lerobot 0.5.1. 3-5 days engineering.
  2. Sequential VLM warm-start (cheap proxy): pretrain just the VLM on VQA pairs in plain HuggingFace transformers, then use the warm VLM as the init for the SmolVLA / Pi0.5 fine-tune. Sidesteps the LeRobot blocker. ~3-6h Brev + 1 day engineering.

Validation status. ✓ 97.6% verbatim from Pi0.5 §I. ✓ 94→74% verbatim from blog (note: only in blog text, not in arXiv paper body text).

M6 · Reference image inlined IN the language token stream

Source. Interleave-VLA (arXiv 2505.02152), §3.2.

Plain-language description. Instead of passing the reference photo as a separate camera observation (which is what we did via camera2), the reference image is inserted INSIDE the language prompt — interleaved between text tokens that reference it. This lets the language model directly ground a noun phrase ("Yann LeCun") to the visual content of the reference image via positional adjacency.

Exact mechanism (paper §3.2 L184-186):

  • The prompt is a sequence $I = (l_1, I_1, l_2, I_2, \ldots)$ — alternating text segments $l_i$ and image tensors $I_i$.
  • Example prompt: "Place [image1] into [image2]." where the bracket is replaced by image tokens at the right position in the text stream.
  • The paper does NOT define specific token strings. Concrete tokens come from the underlying VLM (e.g., PaliGemma's <image> soft tokens). Earlier internal drafts of our research notes invented <BOI>/<EOI> tokens that are not in the paper.

What our current SmolVLA does (code-verified at modeling_smolvla.py:626-705): The prefix order is [img1_tokens, img2_tokens, ..., language_tokens, state_tokens]. Language is appended as a single block AFTER all images. SmolVLA's add_image_special_tokens flag only wraps each individual image with SmolVLM's <global-img> / <end_of_utterance> tokens. It cannot interleave images between text segments.

⚠ Honest caveat The conclusion that the 2nd-camera approach fails compared to the inlined approach is our hypothesis, not paper-tested. Interleave-VLA never tested the 2nd-camera variant head-to-head nor explicitly argued against it. The mechanism finding (interleaved ≠ 2nd-camera) is code-verified; the head-to-head comparison is not.

Empirical impact (Interleave-VLA Table 1, validated):

  • Semantic L1 (novel object, known category): π₀ 26.7 → Interleave-VLA 63.7 (~2.4×)
  • Semantic L2 (novel category): 21.0 → 53.0 (~2.5×)
  • Real FANUC OOD: π₀ w/PT mean ~28% → Interleave-VLA w/PT mean ~58% across 5 OOD object axes (~2×)

Scale caveat. Tested only at ≥3.3B params (π₀-PaliGemma, OpenVLA + InternVL2.5-8B). The phrase "model-agnostic" appears 4× in the paper but with zero sub-1B evidence.

Cost. Multi-day engineering. Requires processor changes in processor_smolvla.py (insert image tokens INTO the text token stream) plus embed_prefix modifications.

Validation status. ✓ Mechanism confirmed by V3 audit (code-verified). ✗ BOI/EOI token notation was fabricated — corrected. ✗ Original Semantic L1 number (30.2→55.7) was wrong; actual is 26.7→63.7 — corrected. ✗ Original Real-FANUC "13→71" was not in Table 2 — corrected to ~28%→~58%. ⚠ 2nd-camera-vs-inlined comparison is our hypothesis.

M7 · 3-5 reference photos per object, sampled per step

Source. Interleave-VLA Table 4.

Plain-language description. Use multiple reference photos per celebrity, sampled randomly at each training step. Mix task-specific crops (from trajectory frames) with internet-source photos of the same person.

Empirical impact (Table 4 verbatim):

Reference sourceScore 1Score 2
Internet-only59.269.1
Task-specific-only67.567.1
Mixed (both sources)71.071.7

Paper §3.3 explicit: cropped-only "lacks diversity," internet-only "lacks task relevance" — combination is required for full OOD gain.

Cost. ~3-4h engineering. Dataset-side change to expand the 192-celeb bank from 1 photo/celeb to 3-5 photos/celeb (we have ≥5 photos for most celebs already) and modify the dataloader to sample one per training step.

Validation status. ✓ Table 4 numbers verified verbatim. Mechanism verified.

M8 · Bbox-grounding co-fine-tune at 10:1 robot:VL

Source. ObjectVLA (arXiv 2502.19250), §3.2-3.3, §4.1.2.

Plain-language description. Mix a vision-language task into the robot fine-tune: given an image of an object, predict the bounding box of that object. The "language" output is the bounding-box coordinates as text. The same bbox prediction is injected as a reasoning prefix in the robot trajectories.

Exact recipe (paper §3.2-3.3 verbatim):

  • VL pair format: (image, "Detecting the bounding box of <object>.", "(x1,y1),(x2,y2)")
  • Robot:VL ratio: 10:1
  • Dataset size: 100 objects × 20 images/object = 2 000 VL pairs
  • Grounding-token injection in robot prompts: <object_ref_start>name<object_ref_end><box_start>(x1,y1),(x2,y2)<box_end> then the action chunk
  • Training: 8× A800, Adam (not AdamW), LR 2e-5 constant, batch 128, 50k steps

Empirical impact (§4.1.2 verbatim):

ConfigurationID successOOD success (100 objects)
ObjectVLA full recipe100%64%
Without bbox-grounding co-train100%19%
Without any VL co-train (DiVLA baseline)—8% (random)

The bbox-grounding co-train carries 45 percentage points of OOD performance. This is the largest single-mechanism gain in the literature we surveyed.

Cost. Two ways:

  1. Full co-training (proper): requires multi-dataset training infra. Same LeRobot blocker as M5. 3-5 days engineering.
  2. Prompt-relabel proxy (cheap): precompute the face bbox per reference photo offline (via RetinaFace), inject it as text in the existing prompts. Example: "<ref> shows Yann LeCun in bbox (12,15)-(245,230). Set the coke down on his picture." ~6h engineering. Weakened version — may or may not transfer.

Validation status. ✓ All numbers verbatim per V4 audit. ✗ Original "AdamW" was wrong — corrected to Adam.

§3 · All 17 training options

Each option combines an architecture × a protocol × a subset of M1-M8.

Option 0 — Ship as-is

Mechanisms: noneBrev: $0Bonus: +20Baseline only

Use HBOrtiz/smolvla_eval3 exactly as-is. No retraining. Empirically fails (positional shortcut). Comparison floor only (~31 pts expected).

Option 1 — SmolVLA + reference recuration only

Mechanisms: M7 (partial)Brev: $0Bonus: +20Add-on

Filter the 192-celeb bank for enrollment-quality photos (NIST FRVT / ISO 19794-5 standards: pose ≤±15°, inter-eye ≥60px, plain background). Pre-crop offline. Ship as static asset table. Pure inference-time fix. Likely +0-2 rollouts alone but free to add to anything.

Option 2 — SmolVLA + print-domain augmentation on camera2

Mechanisms: none (domain coverage)Brev: $15Bonus: +20Add-on

Augraphy-inspired print emulation (Lab gamut → Floyd-Steinberg dither → Perlin grain → blur → JPEG) on camera2 at training time with p=0.7. Re-fine-tune from 30k for 5-10k steps. Direct fix for (G1) domain gap. RoboEngine paper explicitly warns: "Methods even directly modify the scene using random images or texture…fail to respect physical constraints, leading to degenerated real-world performance due to distribution shifts."

Option 3 — SmolVLA + ArcFace cosine distillation only

Mechanisms: M1 + M2Brev: $15Bonus: +20Add-on

Add Blind-VLA's auxiliary cosine loss at λ=0.2. Frozen 3-layer MLP projector. Mid-LLM (Backbone2Enc) injection. Mask-gated to face patches via RetinaFace. ArcFace buffalo_l teacher. Directly fixes (G2) representation gap. Bundle with Options 1 + 2.

Option 4 — VLM-only VQA warm-start (cheap M5)

Mechanisms: M5 (sequential proxy)Brev: $25Bonus: +20Helps everything

Pretrain SmolVLM2-500M on face-VQA pairs (face_crop, "Who is this?", "Yann LeCun") from the 192-celeb bank using plain HuggingFace transformers.Trainer. Sidesteps LeRobot's MultiLeRobotDataset blocker. Then start the SmolVLA fine-tune from the warm VLM. Cheap, works regardless of protocol.

Option 5 — Bbox prompt-relabel (cheap M8)

Mechanisms: M8 (text proxy)Brev: $25Bonus: +20Strongest single-mechanism signal

Rewrite training prompts to include the celebrity's face bounding box in camera2. Precompute bboxes once with RetinaFace. Example new prompt: "<ref> shows Yann LeCun in bbox (12,15)-(245,230). Set the coke down on his picture." Cheapest path to the largest published single-mechanism gain (45pp in ObjectVLA's ablation).

Option 6 — SmolVLA-boost-v2 (Track A v2) PRIMARY

Mechanisms: M1 + M2 + M5-proxy + M7 + M8-proxyBrev: ~$30Bonus: +20Track A

Stack Options 1 + 2 + 3 + 5 plus lower LR to 2.5e-5 plus tighten hue jitter to ±0.02. Resume from 30k for 10-15k more steps. Addresses both domain gap (G1) AND representation gap (G2). Keeps the +20 bonus. Cheapest of the committed tracks.

Option 7 — Option 6 + true Interleave-VLA inlining

Mechanisms: + M6Brev: $30Bonus: +20Follow-up

Option 6 plus modify processor_smolvla.py + embed_prefix in modeling_smolvla.py:626-705 so the prefix becomes [language_with_inlined_images, state] instead of [images, language, state]. Fixes the architectural deviation that V3 audit identified as most likely root cause of v1's failure. 1-2 days engineering. Track A-2 if Option 6 underperforms.

Option 8 — Option 6 + VLM warm-start

Mechanisms: Option 6 + M5Brev: ~$45Bonus: +20Max-effort SmolVLA

Run VLM-only VQA pretrain (Option 4) first, then Option 6 fine-tune from the warm VLM checkpoint. Strongest bonus-preserving combination (5 of 8 mechanisms covered).

Option 9 — Pi0.5 + image-as-prompt (vanilla, Track C)

Mechanisms: none (pure capacity bet)Brev: ~$150Bonus: +16Reconsider

Train lerobot/pi05_base on our existing image-as-prompt dataset. Just swap architectures. PaliGemma vision tower is bigger than SmolVLM's. Strictly weaker than Option 10 for similar Brev cost — recommend dropping in favor of 10.

Option 10 — Pi0.5 + ArcFace distillation (Track B)

Mechanisms: M1 + M2Brev: ~$160Bonus: +16Capacity hedge

Option 9 plus port the Blind-VLA alignment loss recipe to Pi0.5's PaliGemma vision tower. Combines capacity bet (Pi0.5) AND representation fix (ArcFace distill). Strictly stronger than Option 9 for similar cost. Best Pi0.5 option for the budget.

Option 11 — Pi0.5 + true KI recipe

Mechanisms: M3 + M4 + M5 (all full)Brev: ~$160Bonus: +16Skip — eng-blocked

Implement KI's full mechanism: sg(K_b), sg(V_b) in cross-attention, FAST-token CE loss, web/VQA co-train at Pi0.5's mixture ratio. 3-5 days engineering across multiple code paths. Not feasible in deadline.

Option 12 — Pi0.5 + name-only (pre-pivot plan)

Mechanisms: noneBrev: ~$140Bonus: +16Pivot test

Train Pi0.5 with text-only prompts ("Place the coke on Yann LeCun"). No camera2 reference stream. Banks on PaliGemma's WebLI prior. The original team plan before the 2026-05-09 pivot. Epistemically valuable but lower priority than Option 10.

Option 13 — Pi0.5 + name-only + VLM warm-start

Mechanisms: M5 warm-startBrev: ~$165Bonus: +16Best text-only

Option 12 plus VLM warm-start (Option 4) on PaliGemma-3B for celeb-name binding. Highest-capacity text-only option. Best choice if TAs require text-only.

Option 14 — SmolVLA + name-only + VLM warm-start

Mechanisms: M5 warm-startBrev: ~$50Bonus: +20Text-only insurance

Text-only variant of Track A. Drop the camera2 reference stream entirely, name-only prompts, with VQA pretrain on SmolVLM2-500M. Bonus-preserving text-only option.

Option 15 — SmolVLA 3-celeb baseline (Track D) SAFETY NET

Mechanisms: noneBrev: ~$35Bonus: +20Floor

Filter merged dataset to just Swift/Obama/LeCun episodes (178 base teleops). Name-only prompts. Train SmolVLA from lerobot/smolvla_base. --policy.empty_cameras=2. Concedes the 3 OOD eval runs by design but maximises reliability on the 6 IID runs.

Option 16 — Pi0.5 + everything (M1-M8 all-in)

Mechanisms: M1-M8 (all)Brev: ~$170Bonus: +16Skip — eng-blocked

Track A v2's full mechanism stack ported to Pi0.5 + true KI + true Interleave-VLA + true bbox co-train. 3-5 days engineering across 4 code paths. Not feasible.

§5 · Budget scenarios & recommendations

Remaining Brev budget ≈ $130. The committed 4 tracks total ~$370 — over budget. Three concrete scenarios:

Scenario A — Original 4 tracks (~$370, over budget)

Track A (Option 6) + Track B (Option 10) + Track C (Option 9) + Track D (Option 15). Needs either a budget extension or dropping a track.

Scenario B — Drop Track C, keep 3 tracks (~$225)

Track A + Track D + Track B. Rationale: Option 10 (Pi0.5+ArcFace) is strictly stronger than Option 9 (Pi0.5 vanilla) for similar Brev cost. We lose the "pure capacity test" but gain a track that includes a surgical fix. Still ~$95 over the remaining $130 budget.

Scenario C — Minimum viable 2 tracks (~$65, fits budget)

Track A (Option 6) + Track D (Option 15). Both bonus-preserving. No Pi0.5 capacity test. Mitigation: add Option 4 (VLM warm-start) at ~$25 — boosts both A and D, total $90 still under budget.

Scenario D — Text-only branch (if TAs reject image-as-prompt)

Option 14 (SmolVLA + name-only + VLM warm-start) + Option 15 (Track D) + optionally Option 13 (Pi0.5 + name-only + VLM warm-start). Triggered by the TA permission question (§6).
