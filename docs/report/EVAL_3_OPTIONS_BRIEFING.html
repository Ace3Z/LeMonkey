LeMonkey Eval 3 — Text-Only Pivot

v3 strategy after TA ruling on image-as-prompt · ETH Robot Learning FS26 Project 1 · 2026-05-18
⚖ TA ruling — 2026-05-18, 08:38

Question asked (R. Zninvo, LeMonkey team):

For Eval 3, can our policy take a reference image of the named celebrity as a 2nd camera input (looked up internally from a pre-built name→photo asset table that ships with our policy)? Or must the policy take only the text prompt as input, with no auxiliary image lookups? There is no face matching or ArcFace stuff going on. We just give a reference photo of the same celeb as input in hopes that it might help the VLA to reason better.

Carl Brander's reply:

the idea is to keep it end-to-end and general as this is the current struggle you would also encounter in industry (trying to make a robot policy that is general enough to not rely on task-specific inputs) so we encourage (and enforce) you to do vanilla VLAs with a scene camera + text (+proprioception) as input only. But maybe you want to look into finetuning your VLM on celebrity image recognition and then use this VLM as a VLA on your task. This should work just as well. — Carl Brander, TA

Verdict. Policy input is restricted to scene_camera + text + proprioception. The reference-photo channel (observation.images.reference) we shipped in our v1/v2 pipeline is disallowed. Carl's suggested alternative — VLM finetune on celebrity recognition before VLA training — maps directly to our pre-enumerated M5 (VLM warm-start) mechanism and becomes the new load-bearing addition.

What this document covers This is the post-ruling pivot doc, replacing the earlier image-as-prompt-centric briefing. It (1) re-summarises which of our 8 mechanisms (M1–M8) survive the ruling, (2) lays out the new Track 1 / 2 / 3 v3 plan, (3) details the impact on the in-flight Track 3 augmentation, and (4) lists implementation tasks. Background and the original M1–M8 mechanism derivations are at EVAL_3_RESEARCH_REPORT.md; previous training-plan iterations are in eval_3/STRATEGY.md §7a–§7c (now superseded by §7d, being written).

§1 · Impact of the ruling

The Eval 3 architecture decision locked on 2026-05-09 — "Path A, image-as-prompt" — required passing a reference photo of the target celebrity as a camera2 stream alongside the wrist camera. Every strategy iteration since then has built on this protocol. Carl's ruling invalidates the architecture choice.

§2 · M1–M8 status post-ruling

The eight mechanisms we enumerated for surgical SmolVLA-450M training. Of these, three are training-only-no-inference-input and survive trivially; one is now load-bearing (M5); two die.

M1 — Frozen 3-layer MLP projector · ALIVE

Internal training infra for ArcFace distillation. No inference input. Source: BlindVLA arxiv 2510.25616 Table 6. Status unchanged.

M2 — ArcFace cosine alignment at SmolLM2 Backbone2Enc layer 7/8 · ALIVE — now critical

$$ L_{\text{align}} = -\frac{1}{k} \sum_{j=1}^{k} \cos\!\left(F.\text{normalize}\!\left(\text{MLP}(h_l)_j\right), F.\text{normalize}(z_j)\right), \quad \lambda = 0.2 $$

Where $h_l$ is the SmolLM2 hidden state at the Backbone2Enc layer (default $l=8$), $z$ is the cached buffalo_l ArcFace embedding of the face visible in the wrist camera (precomputed offline), and the MLP is frozen after init. Camera1-only. The teacher embeddings are stored as a static cached table referenced by face-region; they are not retrieved at inference. The distillation forces SmolLM2 to develop face-discriminative features from the wrist-camera image alone, which is exactly the skill the model now needs without a reference channel.

Source: BlindVLA arxiv 2510.25616 §6.2 Eq. 9. Status: load-bearing for face identity recovery from camera1 alone.

M3 — Pi0.5-KI stop-gradient on $K_b$, $V_b$ · ALIVE

In the action expert's cross-attention to the VLM, both key and value matrices from the VLM are wrapped in sg(·): $K_b = \mathrm{sg}(K_{\text{vlm}})$, $V_b = \mathrm{sg}(V_{\text{vlm}})$. Prevents action-loss gradient from corrupting the VLM. Pure gradient-flow control — no input change. Source: Pi0.5-KI arxiv 2505.23705 Eqs. 5–6. Status unchanged.

M4-lite — FAST-token CE loss on VLM LM head, $\lambda=0.1$ · ALIVE

Training-only loss. Per our 2026-05-18 Validation #1, M3-without-M4 leaves upper SmolLM2 layers (8–15) without task gradient; Pi0.5-KI Fig. 6(b) shows 7.5× slower convergence without FAST CE. λ=0.1 is the recommended mitigation strength. Source: Pi0.5-KI §3.3. Status unchanged.

M5 — VLM warm-start on celebrity VQA · PROMOTED — now load-bearing

Before VLA training, fine-tune the SmolVLM2-500M backbone on a name→face VQA task. Carl's recommendation maps to this mechanism verbatim. Training-only; inference is still text-only.

See §5 deep dive for the data recipe + script outline.

Source: Pi0.5 blog (Mar 2025) "Web/VQA co-training preserves VLM priors during robot fine-tune"; Carl Brander 2026-05-18 ("finetuning your VLM on celebrity image recognition and then use this VLM as a VLA"). Status: promoted from deferred to mandatory.

M6 — Interleave-VLA inline reference image in language stream · DEAD

The mechanism splices a reference photo's image embedding into the language token stream at a <image> marker, giving the LM positional grounding for the named celebrity. Requires a reference image at inference. Forbidden by ruling.

Implementation we already shipped (commit b50b76a) is parked: the m6_inline_reference config flag defaults to False, so the patch is benign when present. We keep the code as a reference implementation for future projects but do not activate it for Eval 3.

M7 — 3–5 reference photos per celeb, sampled per training step · DEAD

Designed to vary the reference channel across training steps so the model doesn't memorise a specific photo. Becomes moot once the reference channel itself is gone. Forbidden by ruling.

M8 — ObjectVLA-style bbox-grounding co-train · ALIVE (deferred)

Auxiliary task: teach the model to predict the bounding box of the named celeb's face in the wrist-camera view (e.g. via a text-output head producing "(x1,y1)-(x2,y2)"). Operates on camera1 only. No reference channel needed. Source: ObjectVLA arxiv 2502.19250 §4.1.2 (reports 100% ID / 19% OOD without bbox grounding; 100% → 64% with). Status: still deferred (multi-day implementation cost); flag for follow-up if Track 1 v3 underperforms.

§3 · New Track 1 / 2 / 3 v3 plan

All three tracks share the same input contract: observation.images.camera1 (wrist) + observation.state + text prompt naming the target celeb. No reference image stream. --policy.empty_cameras=1 zero-pads the second camera slot if the dataset schema still carries it.

Track 1 v3 — SmolVLA-450M + M1 + M2 + M3 + M4-lite + M5

backbone: HuggingFaceTB/SmolVLM2-500M-Video-Instruct → lerobot/smolvla_base input: camera1 + state + text only brev: ~6 h after warm-start bonus: +20 (smallest-model)

Stage 1 (M5 warm-start, ~6 h Brev) — VLM-only fine-tune. Start from SmolVLM2-500M-Video-Instruct on Hugging Face. Build a small VQA dataset: ~30 celebrities × ~50 photos each (scraped, face-quality-filtered) + Swift / Obama / LeCun from our existing bank. Each row is (image, question="Who is in this photo?", answer=""). Run autoregressive LM loss for ~5k–10k steps. Push the warm-started VLM to HBOrtiz/smolvlm2_celeb_warm.

Stage 2 (VLA fine-tune, ~6 h Brev) — action training. Initialize SmolVLA with the warm-started VLM weights from Stage 1. Continue with the full M1+M2+M3+M4-lite stack:

$$ L_{\text{total}} = L_{\text{flow-matching}} \;+\; 0.1 \cdot L_{\text{FAST-CE}} \;+\; 0.2 \cdot L_{\text{align}} $$

Gradient flow: M3 stop-gradient on $K_b$ / $V_b$ from VLM into action expert (Pi0.5-KI Eqs. 5–6); M1 frozen MLP projector (BlindVLA Table 6).

Dataset: 178 base teleops + 9,216 augmented variants from Track 3's aug pipeline (camera1 stream only; ref stream ignored).

Checkpoint: push to HBOrtiz/smolvla_eval3_track1_v3.

Track 2 v3 — Pi0.5-3B + M1 + M2 + M3 + M4-lite + M5

backbone: google/paligemma2-3b → lerobot/pi05_base input: camera1 + state + text only brev: ~32 h after warm-start bonus: +16

Stage 1 (M5 warm-start, ~10 h Brev). PaliGemma2 already has strong celebrity priors from its WebLI pretraining (π₀.5 blog notes 94% → 74% OOD degradation without web co-training). Light VQA fine-tune on our 30-celeb dataset to anchor the names → faces in the new task distribution.

Stage 2 (VLA fine-tune, ~32 h Brev). Mirror Track 1's mechanism stack on PaliGemma. M2 injection layer: PaliGemma layer 10–12 (mid-depth of 18). M3 is the native deployment of Pi0.5-KI's protocol.

Brackets the failure-mode hypothesis. If Track 2 ≫ Track 1, capacity is load-bearing and the smallest-model bonus (+4 differential) is worth giving up. If Track 2 ≈ Track 1, surgery alone is enough and we ship SmolVLA for the +4.

Checkpoint: push to HBOrtiz/pi05_eval3_track2_v3.

Track 3 v3 — SmolVLA 3-celeb baseline (safety floor)

backbone: lerobot/smolvla_base (no warm-start) input: camera1 + state + text only brev: ~6 h bonus: +20

The most important thing that must work. Concedes the 3 OOD runs by design (training distribution restricted to Swift / Obama / LeCun) but maximises reliability on the 6 IID runs.

Composition:

  • Backbone: lerobot/smolvla_base (no M5 warm-start — fastest path).
  • No M1 / M2 / M3 / M4-lite. Pure vanilla SmolVLA with heavy data diversity.
  • Dataset: 178 base teleops + 9,216 augmented variants (camera1 only).
  • Prompts: 100% default bucket — "Place the coke on {NAME}." No counterfactual, no ref-only (those were valid only when a reference channel existed). Post-process pass over the sidecars sets prompt from target_celeb_name.

Worst case: 0 / 3 OOD + 6 / 9 IID + 20 bonus = (6/9) · 50 + 20 ≈ 53 pts. Best case: 0 / 3 OOD + 9 / 9 IID + 20 bonus = 50 + 20 = 70 pts.

Checkpoint: push to HBOrtiz/smolvla_eval3_track3_v3_baseline.

§4 · What happens to the running Track 3 augmentation?

As of 2026-05-18 12:36 CEST: 4,913 / 9,216 variants on disk (53.3 %), 4 parallel workers, 8.93 variants/min steady rate, ETA ~20:30 CEST tonight. The dataset is mid-flight; the ruling arrived after launch.

Decision: let it finish. Per-variant cost is dominated by the wrist-camera (camera1) compositing — 538 frames × 3 portraits × OpenCV warp + Reinhard + seamless clone — which is still the visual data the model now has to learn identity from. The reference video stream is written cheaply at the end (~150 KB constant-frame mp4) and is wasted at training but the disk cost is negligible (~1.4 GB total). No CPU saving from killing it now (we'd lose ~50 % of the camera1 work).

Post-aug clean-up tasks (cheap, no recompute):

  1. Prompt re-label pass. Walk all 9,394 episode dirs (178 base + 9,216 aug). For each, rewrite reference.json["prompt"] to the default bucket using target_celeb_name already in the sidecar: 
    prompt = random.choice(DEFAULT_PARAPHRASES).format(name=sidecar["target_celeb_name"])
    Drop the ref_only and counterfactual buckets entirely. ~10 minutes of disk-walk time.
  2. Training-script input contract. Set --policy.empty_cameras=1 (or omit the reference key from input_features) so the second camera slot is zero-padded and ignored by the model. No model code change.
  3. HF push. Merge to LeRobot v3 format, push to HBOrtiz/so101_eval3_track3_v3_baseline.

§5 · Deep dive — M5 (VLM warm-start on celeb VQA)

Carl's recommendation: "finetuning your VLM on celebrity image recognition and then use this VLM as a VLA on your task. This should work just as well." Below is the concrete recipe.

5.1 · Data recipe

Goal. Teach the VLM that given a celebrity face crop, the right text output is the celebrity's name. Standard VQA format.

Bank construction.

  • ~30 celebrities total: Swift / Obama / LeCun (the 3 IID identities) + ~27 representative public figures we expect to be useful generic priors (could include the additional names already in the task spec's OOD pool, plus famous-by-default people that PaliGemma / SmolVLM2 likely already know).
  • ~50 photos per celeb, scraped + face-detector-filtered (det_score ≥ 0.7, single face, no group shots). Already have eval_3/scripts/scrape_headshots.py for this.
  • Crop policy: head + shoulders, ~30 % padding around the InsightFace bbox. Matches what the wrist camera sees of a printed paper at training time.
  • Augmentations at fine-tune time: random crop, brightness ±20 %, slight blur (σ ≤ 1 px) — small distribution shift to prepare the VLM for the wrist-camera printed-photo style at VLA-eval time.

VQA format. Each row is one of three templates, drawn uniformly:

  1. question: "Who is in this photo?" answer: "{name}"
  2. question: "Identify the person in the image." answer: "This is {name}."
  3. question: "Is this {name}?" answer: "Yes." (50 %) or "No, this is {other_name}." (50 %) — adds discriminative pressure.

5.2 · Training recipe (SmolVLM2-500M side, Track 1 v3)

  • Optimizer: AdamW, lr = 2.5e-5, weight decay 1e-2.
  • Batch size 16, grad accumulation × 4 (effective 64).
  • Steps: ~5k (1 epoch over 30 celebs × 50 photos × 3 question templates = ~4500 rows; small overshoot for paraphrase coverage).
  • Loss: standard autoregressive CE on the answer tokens (mask out the question / system tokens).
  • Vision encoder: freeze SigLIP. Only the SmolLM2 layers and the connector update. Mirrors SmolVLA's published recipe and keeps the warm-started weights aligned with the action-training expectation.
  • Estimated wall time: ~4 h on the dev box's RTX 3090, ~3 h on Brev's RTX PRO 6000 Blackwell.

Output: the fine-tuned VLM state-dict. Save under HBOrtiz/smolvlm2_celeb_warm/. Load this as --policy.vlm_pretrained_path when launching SmolVLA training (SmolVLAConfig.vlm_model_name already supports a custom path; no code change needed).

5.3 · Why this should help

Two effects combine:

  1. Direct name → face binding. The VLM emerges from warm-start knowing that the visual features of celebrity X correspond to the text token sequence "X". At VLA training time, the action expert (which conditions on the VLM's hidden state) gets a richer signal when the wrist camera sees a printed face of X.
  2. VLA training stability. Without warm-start, the model has to learn name → face and face → action simultaneously from 178 + 9,216 ≈ 9,400 demos — and the v1 training showed that 178 demos at 450M scale weren't enough to learn the name-binding component. Pre-binding decouples the two skills.

This is consistent with Pi0.5's reported 94 % → 74 % OOD degradation when web/VQA co-training is dropped — the VLM's celebrity priors are the missing ingredient.
