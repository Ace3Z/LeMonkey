The Name → Face Binding Is Broken

Eval 3 attention-routing diagnosis · LeMonkey · 2026-05-20

Across every model we trained — SmolVLA and Pi0.5 alike — the robot picks up the can and places it smoothly, but lands it on the wrong celebrity. This document isolates the single mechanism responsible, with a controlled probe, and states the one intervention that addresses it head-on.

1. The probe

We ran a controlled, single-variable test. Hold the scene completely fixed — the same three printed portraits, the same image. Vary only the celebrity name in the text prompt:

Swift  →  Obama  →  LeCun

Then inspect the VLM's cross-modal attention — specifically, where the name token attends among the image patches.

Result

The name-token → image attention does not change. It parks on the same fixed background patch every time — an "attention sink" in a corner of the image — regardless of which celebrity is named. It never lands on the named celebrity's face.

2. Why this is the issue

The task is "go to the named celebrity." The only mechanism by which a name can pick out a face is cross-modal attention — the name token must look at the image patch where that celebrity's face is. The intended chain is:

name token "Obama"
✗→
cross-attention routes to Obama's face patch
→
VLM output name-conditioned
→
action expert moves to the right printout

The red link is the broken one. If the attention is name-invariant, the name is never grounded to a face. The action expert downstream reads the VLM's output; with no name-conditioned difference in that output, the policy has no signal for which printout to move to — so it falls back to a positional guess. That one missing link is the entire task.

3. Why it is the cause, not a symptom

EvidenceWhat it rules out
The probe isolates it. Same image, only the name varies — and the attention still doesn't move. Not a data, prompt, or scene-rendering problem. The single varied input fails to drive the single relevant mechanism.
Identical failure on every model. SmolVLA Track D (whose Strix rollout failed — the behavioural symptom), warm-VLM v1, and warm-VLM v2 all show the same name-invariant attention. Not a one-off checkpoint artifact. It is structural across backbones and recipes.
Good representations did not fix it. M2 (ArcFace distillation) shaped the face-crop representations well — mean cosine 0.88 on crops — and the routing still did not move. Rules out "representation quality" as the gap. The faces are well-encoded; the name still can't route to them.
Diagnosis

The gap is specifically attention routing — the name → face patch binding — not representation quality and not a downstream action problem. Every prior intervention improved something else and left this link untouched.

4. Why representation-only fixes can't close it

A face-VQA naming loss (single face image → name) improves representation and naming — but it does not directly supervise routing in a multi-face scene. In a VQA example there is only one face, so there is no "which patch" decision to learn. That is exactly why representation-only interventions like M2 didn't close the gap: they were never training the thing that's broken.

5. The intervention: supervise the routing directly (KLAL)

KLAL directly supervises the name → face attention with bbox-derived targets. For each training frame we already know where every celebrity's face is — from the augmentation metadata, with RetinaFace as a fallback. We build a target attention distribution that places mass on the named celebrity's face patches, and penalize the divergence between the model's actual name-token → image attention and that target.

It is the only mechanism on the table that addresses the diagnosed failure head-on, rather than improving an adjacent quantity.

Refinement 1 — KLAL needs a trainable, identity-capable vision tower to route onto

On a frozen, identity-blind vision tower (the v1 condition — cosine 0.87 same-person vs 0.82 different-person, near-zero identity separation) there is no identity-discriminable patch for KLAL to route the name token to, so KLAL cannot converge. KLAL must run with the vision tower trainable (the v2 condition).

The upside: KLAL's gradient, flowing back through a trainable vision tower, itself drives the tower toward identity-discriminable features — because making the name token attend to the correct face among three is only possible if the patches encode identity. So KLAL + a trainable vision tower does double duty: it supervises routing and shapes the representation that routing depends on.

Refinement 2 — KLAL must be co-trained with the action loss, not run sequentially

A KLAL-then-action-finetune sequence would drift exactly like the v1/v2 warm-starts did — an action-only second stage has nothing keeping the routing graded, so the skill decays. The run must be a single co-training pass: action loss + KLAL, both alive every step, vision tower trainable. KLAL fixes the routing; the co-trained action loss lets the action expert learn to use the now name-conditioned VLM output.

Relationship to ObjectVLA

ObjectVLA's bbox-grounded captions ("the face of X is at [bbox]" fed as text) are the indirect version of the same idea — they make the model learn spatial grounding by parsing bbox coordinates out of text. KLAL is the direct version: it shapes the attention map itself. KLAL is the more surgical of the two; ObjectVLA-style bbox captions can serve as a cheap add-on, but KLAL is the primary mechanism.

Success metric. The verdict is the rollout — does the arm place the can on the correct celebrity — not the attention map looking correct. Attention supervision is the mechanism; the rollout is the test. Track both, but ship on the rollout.

6. Implementation sketch

  • Per training frame: face bounding boxes for all three portraits — known from our augmentation metadata; RetinaFace as fallback.
  • Map each bbox (normalized coords) → the 16×16 PaliGemma image-token grid (256 tokens) → the image-token indices covering that face.
  • During the forward pass, capture the cross-attention weights at the chosen layer(s): rows = name-token positions, columns = image-token positions.
  • Target distribution: mass on the named celebrity's face-token indices.
  • Loss: KL divergence between model attention and target, summed over the chosen layers/heads — added to the total loss alongside the action loss.
# co-training loop — one run, two losses alive every step
for step in batches:
    action_loss = pi05_flow_matching_loss(robot_batch)
    klal_loss   = kl_div(name_token_attention, bbox_target)   # the new term
    loss = action_loss + lambda_klal * klal_loss
    loss.backward()        # vision tower trainable — gradient shapes it too
    optimizer.step()

Estimated ~200–300 lines hooking into lerobot's Pi0.5 per-layer attention loop.

Bottom line

The probe pins the failure to one link: the name token never attends to the named face. Representation-quality fixes (M2, ArcFace) cannot close it because they don't train routing. The fix is to supervise the routing directly — KLAL — co-trained with the action loss, on a trainable vision tower. That is the next run.
